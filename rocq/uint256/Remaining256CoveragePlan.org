* Completed Proof Note: `signextend_correct`

`signextend_correct` now follows the refactored Rocq model in
[[theories/Arithmetic.v][theories/Arithmetic.v]], not the older
four-constructor proof attempt. The implementation reads:

1. guard on `byte_index_256 < 31`
2. collapse to `byte_index := w0 byte_index_256`
3. compute `word_index := shr byte_index 3`
4. translate that word index to `nat`
5. read the selected word with `get_word`
6. transform it with `signextend_current_word`
7. write it back with `set_word`
8. fill the suffix with `fill_words_from`

Completed proof structure:

1. Split on the outer `ltb_uint256` guard.
   - false branch: prove `to_Z_uint256 byte_index_256 >= 31`, so
     `signextend_Z` reduces to the identity on `x`
   - true branch: derive `to_Z_uint256 byte_index_256 < 31`

2. In the true branch, show the high words of the byte index are zero.
   Use `to_Z_uint256_lt_base_zero_high` after observing that
   `31 < base width`, then rewrite everything in terms of `b0`.

3. Normalize the local indices:
   - `word_index = to_Z b0 / 8`
   - `bit_index = 8 * (to_Z b0 mod 8)`
   - `bits = 8 * (to_Z b0 + 1)`
   - `0 <= to_Z (shr b0 3) < 4`

4. Rewrite `signextend_word_index_nat` using
   `signextend_word_index_nat_correct`, then case-split on the resulting
   `nat` index `0`, `1`, `2`, `3`.

5. For each case, rewrite the concrete writeback shape using the new
   `signextend_writeback_word{0,1,2,3}` lemmas. This should collapse the
   `set_word` / `fill_words_from` pipeline to the same four visible
   result shapes as the C++ control flow.

6. On the selected word, apply `signextend_current_word_correct` and
   finish by matching the preserved prefix and filled suffix against the
   `signextend_Z` target.

* Proof Worklog: `signextend`

Record what materially helps and what does not during the live proof
effort. Update this section as the proof progresses.

What helps:

- `rocq_start` and `rocq_check` now return both `goals` text and
  `structured_goals` on the live
  [[theories/ArithmeticProofs.v][theories/ArithmeticProofs.v]]
  proof. The structured payload is useful for reading the exact
  hypothesis list after each branch split without scraping pretty
  text.
- Small scratch goals in `rocq-mcp` help validate explicit rewrite
  sequences before transplanting them into the real proof. This is
  much cheaper than rebuilding the whole file after every failed
  algebraic rewrite.
- The refactored `signextend` model is easier to reason about than the
  old constructor split. Once `signextend_word_index_nat` is rewritten,
  each index case should reduce quickly through the writeback lemmas
  instead of carrying `mk_uint256` case analysis through the whole
  theorem.
- In the `word_index = 2` branch of `signextend_correct`,
  `structured_goals` was useful for spotting that the failure at the
  old line-4549 rewrite was a shape mismatch, not a missing arithmetic
  fact. The remaining goal still mentioned `w2 val` and `w3 val`,
  while `Hcur`, `Hsign`, `Hmod2`, and `Hcond2` were stated over `x2`
  and `x3`. The fix is to `unfold val` in those hypotheses and in the
  goal before attempting the rewrite.
- Clearing stale `_build/.lock` files before a rebuild matters. A
  hanging `dune build` without a live `rocq` process was sometimes a
  lock problem, not a proof-performance problem.

What does not help:

- First-pass uses of `nia`, `lia`, and `ring` in this proof are too
  expensive. They obscure which arithmetic fact is actually needed and
  can push the Rocq process into long runs or OOM failure.
- The current `structured_goals` payload is not by itself a complete
  pretty-printer. On live runs the top-level `goals` field is still
  the most readable source of rendered proof state because per-goal
  `pretty` text is often absent.
- `rocq_step_multi` is not reliable enough to treat as the primary
  exploration tool in this host session. Even with the tool enabled in
  local MCP config, the host safety layer can cancel calls, so the
  dependable path is still `rocq_check`.
- `rocq_compile_file` is not a trustworthy gate for this repo at the
  moment because the helper still tries to invoke `coqc` directly.
  The actual build gate remains `opam exec -- dune build`.
- Rebuilding the whole file to debug one local rewrite is too coarse.
  It gives slow feedback and makes it harder to distinguish a real
  proof error from infrastructure issues like stale locks.
- Large monolithic proof branches are brittle. Even when they are
  logically correct, long arithmetic subproofs are hard to debug and
  easy to make slow.

Current working rule:

- First pass: prefer explicit rewrites, local assertions, and short
  `rocq-mcp` scratch checks. Use decision procedures only after the
  shape of the subgoal is stable and the exact missing arithmetic fact
  is understood.

* Shared Rules

1. Start from the C++ function and keep the Rocq definition
   structurally parallel.
2. Preserve control flow, not just extensional semantics.
   Use the same branch order, threshold constants, local names, and
   helper decomposition where practical.
3. Put the concrete wrapper in
   [[theories/Arithmetic.v][theories/Arithmetic.v]]
   only if it is genuinely part of the 256-bit arithmetic surface.
   Put byte and serialization utilities in a separate module if they
   are not arithmetic.
4. For every port, add a correctness theorem in the matching proof
   file before moving on.
5. In the proof, state the semantic target so that a developer can
   compare it directly with the C++ observable behavior.
6. Do not "improve" the Rocq control flow into a more abstract version
   if that makes side-by-side comparison harder.

* Part 1: Immediate Work That Can Be Added Or Finished Now

1. Add thin concrete 256-bit division wrappers.

   Even though `udivrem` is already modeled, add the concrete
   `uint256`-level facade for `udivrem`, `/`, and `%` if API parity
   with `uint256.hpp` is desired. Reuse the existing division model and
   proofs rather than re-proving the algorithm.

2. `byte_correct` is complete.

   The model in [[theories/Arithmetic.v][theories/Arithmetic.v]] follows the
   C++ implementation:

   - out-of-range branch
   - big-endian byte index reversal
   - word selection
   - bit shift
   - low-word result

   The proof-side semantic target is `byte_Z` in
   [[theories/Arithmetic.v][theories/Arithmetic.v]]. The proof
   reuses the `signextend_correct` setup pattern: collapse the bounded
   index to `b0`, normalize `31 - b0`, split on the selected word
   index, and show the masked shift equals the selected byte of the
   256-bit value.

3. `countl_zero_uint256` and `count_significant_bytes` are complete.

   The models in [[theories/Arithmetic.v][theories/Arithmetic.v]] are natural
   wrappers over the existing word-level `countl_zero` and generic
   `count_significant_words`. `countl_zero_uint256_correct` proves the
   high-to-low scan shape, and `count_significant_bytes_correct`
   proves the byte count is the ceiling of the significant bit count
   derived from `countl_zero_uint256`.

4. `byteswap` is complete.

   The model in [[theories/Arithmetic.v][theories/Arithmetic.v]] keeps the
   same reverse-word structure as the C++ initializer and uses a
   word-level helper for the `std::byteswap` operation.
   `byteswap_correct` states the 256-bit result as the reversed list of
   word-level byte-swaps.

5. Record-level bitwise wrappers are complete.

   The wrappers in [[theories/Arithmetic.v][theories/Arithmetic.v]] mirror the
   C++ macro-generated `&`, `|`, and `^` operators and the unary `~`
   operator by applying the corresponding word operation at indices
   `0` through `3`. Their correctness theorems in
   [[theories/ArithmeticProofs.v][theories/ArithmeticProofs.v]] state the
   exact word-list result shape.

6. `countr_zero` and `popcount` are complete.

   The models in [[theories/Arithmetic.v][theories/Arithmetic.v]] preserve the
   C++ low-to-high word order. `countr_zero` accumulates a word-level
   trailing-zero count and returns early when the selected word is not
   all zero. `popcount` sums the four word-level population counts.
   The proof file records bounds for the word helpers, proves the
   `countr_zero` result contains the accumulated count as a `uint256`,
   and states the exact four-word sum shape for `popcount`.
