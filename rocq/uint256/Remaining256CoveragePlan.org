* Immediate Proof Prep: `signextend_correct`

The proof scaffolding is already present in
[[/home/cmcl/projects/category-labs/rocq-u256/rocq/uint256/theories/ArithmeticProofs.v][theories/ArithmeticProofs.v]] immediately above the theorem. The next
`rocq-mcp` pass should be organized around the existing local lemmas
rather than starting from scratch.

Recommended branch structure:

1. Split on the outer `ltb_uint256` guard.
   - false branch: prove `to_Z_uint256 byte_index_256 >= 31`, so
     `signextend_Z` reduces to the identity on `x`
   - true branch: derive `to_Z_uint256 byte_index_256 < 31`

2. In the true branch, show the high words of the byte index are zero.
   Use `to_Z_uint256_lt_base_zero_high` after observing that
   `31 < base width`.

3. Reduce the selected-word logic to four cases.
   After the high-word collapse, split on `word_index = shr b0 3`
   corresponding to bytes `0..7`, `8..15`, `16..23`, and `24..30`.

4. For the selected word, rewrite the local `current_word` result to
   the `signextend_Z` target using the existing signed-byte lemmas.

Primary lemmas already in the file for this proof:

- `ltb_uint256_correct`
- `bounded_shift_nat_correct`
- `to_Z_shr`
- `to_Z_low_mask`
- `to_Z_land_low_mask`
- `to_Z_asr_63`
- `signextend_signed_byte64_correct`
- `modulus_words_1`, `modulus_words_2`, `modulus_words_3`
- `modulus_words_scale_mod`

The useful target normalization is:

- `bits = 8 * (to_Z_uint256 byte_index_256 + 1)`
- `word_index = to_Z_uint256 byte_index_256 / 8`
- `bit_index = 8 * (to_Z_uint256 byte_index_256 mod 8)`

That puts the proof in the same branch order as the implementation and
keeps the end-state comparison with the C++ `signextend` readable.

* Proof Worklog: `signextend`

Record what materially helps and what does not during the live proof
effort. Update this section as the proof progresses.

What helps:

- `rocq_start` and `rocq_check` now return both `goals` text and
  `structured_goals` on the live
  [[/home/cmcl/projects/category-labs/rocq-u256/rocq/uint256/theories/ArithmeticProofs.v][theories/ArithmeticProofs.v]]
  proof. The structured payload is useful for reading the exact
  hypothesis list after each branch split without scraping pretty
  text.
- `rocq_step_multi` is now safer to use for branch exploration because
  its result entries also include `structured_goals`. This is useful
  for checking whether a candidate rewrite or `set` introduces the
  expected local context before committing it with `rocq_check`.
- Small scratch goals in `rocq-mcp` help validate explicit rewrite
  sequences before transplanting them into the real proof. This is
  much cheaper than rebuilding the whole file after every failed
  algebraic rewrite.
- In the `word_index = 2` branch of `signextend_correct`,
  `structured_goals` was useful for spotting that the failure at the
  old line-4549 rewrite was a shape mismatch, not a missing arithmetic
  fact. The remaining goal still mentioned `w2 val` and `w3 val`,
  while `Hcur`, `Hsign`, `Hmod2`, and `Hcond2` were stated over `x2`
  and `x3`. The fix is to `unfold val` in those hypotheses and in the
  goal before attempting the rewrite.
- Clearing stale `_build/.lock` files before a rebuild matters. A
  hanging `dune build` without a live `rocq` process was sometimes a
  lock problem, not a proof-performance problem.

What does not help:

- First-pass uses of `nia`, `lia`, and `ring` in this proof are too
  expensive. They obscure which arithmetic fact is actually needed and
  can push the Rocq process into long runs or OOM failure.
- The current `structured_goals` payload is not by itself a complete
  pretty-printer. On live runs the top-level `goals` field is still
  the most readable source of rendered proof state because per-goal
  `pretty` text is often absent.
- `rocq_step_multi` is not reliable enough to treat as the primary
  exploration tool in this host session. Even with the tool enabled in
  local MCP config, the host safety layer can cancel calls, so the
  dependable path is still `rocq_check`.
- `rocq_compile_file` is not a trustworthy gate for this repo at the
  moment because the helper still tries to invoke `coqc` directly.
  The actual build gate remains `opam exec -- dune build`.
- Rebuilding the whole file to debug one local rewrite is too coarse.
  It gives slow feedback and makes it harder to distinguish a real
  proof error from infrastructure issues like stale locks.
- Large monolithic proof branches are brittle. Even when they are
  logically correct, long arithmetic subproofs are hard to debug and
  easy to make slow.

Current working rule:

- First pass: prefer explicit rewrites, local assertions, and short
  `rocq-mcp` scratch checks. Use decision procedures only after the
  shape of the subgoal is stable and the exact missing arithmetic fact
  is understood.

* Shared Rules

1. Start from the C++ function and keep the Rocq definition
   structurally parallel.
2. Preserve control flow, not just extensional semantics.
   Use the same branch order, threshold constants, local names, and
   helper decomposition where practical.
3. Put the concrete wrapper in
   [[/home/cmcl/projects/category-labs/rocq-u256/rocq/uint256/theories/Arithmetic.v][theories/Arithmetic.v]]
   only if it is genuinely part of the 256-bit arithmetic surface.
   Put byte and serialization utilities in a separate module if they
   are not arithmetic.
4. For every port, add a correctness theorem in the matching proof
   file before moving on.
5. In the proof, state the semantic target so that a developer can
   compare it directly with the C++ observable behavior.
6. Do not "improve" the Rocq control flow into a more abstract version
   if that makes side-by-side comparison harder.

* Part 1: Immediate Work That Can Be Added Or Finished Now

1. Add thin concrete 256-bit division wrappers.

   Even though `udivrem` is already modeled, add the concrete
   `uint256`-level facade for `udivrem`, `/`, and `%` if API parity
   with `uint256.hpp` is desired. Reuse the existing division model and
   proofs rather than re-proving the algorithm.

2. Finish `signextend_correct`.

   The implementation is already in place. The remaining work is the
   correctness proof in
   [[/home/cmcl/projects/category-labs/rocq-u256/rocq/uint256/theories/ArithmeticProofs.v][theories/ArithmeticProofs.v]].
   Keep the proof branch-for-branch with the current Rocq definition:

   - `byte_index >= 31`
   - compute `word_index`
   - compute `bit_index`
   - reconstruct the selected word
   - fill upper words with the sign bits

   Prefer a proof that still reads like the C++ algorithm, even if
   some internal bit twiddling uses Rocq shift identities.

3. Port `byte`.

   Follow the C++ implementation exactly:

   - out-of-range branch
   - big-endian byte index reversal
   - word selection
   - bit shift
   - low-word result

   This is a good dependency for later byte-oriented helpers.

4. Port `countl_zero(uint256)` and `count_significant_bytes`.

   These are natural wrappers over the existing word-level
   `countl_zero` and generic `count_significant_words`. Keep the same
   scan order as the C++ loops.

5. Port `byteswap` after `byte`.

   If close correspondence is the priority, implement it in the same
   reverse-byte-order style as the C++, not via a more abstract
   normalization theorem.

* Part 2: Lower-Level Extensions Needed For The Remaining Clean Ports

1. Extend the word interface in
   [[/home/cmcl/projects/category-labs/rocq-u256/rocq/uint256/theories/Uint.v][theories/Uint.v]]
   with:

   - `and`
   - `xor`
   - `not`

   Add `to_Z` specifications alongside the existing `spec_or`. This is
   the main missing interface piece.

2. Decide whether to add 64-bit utility primitives too:

   - `countr_zero`
   - `popcount`
   - possibly `byteswap`

   These are not strictly impossible without new primitives, but
   adding them makes the Rocq control flow much closer to the C++ and
   much easier to audit side-by-side.

3. After the interface extension, add record-level bitwise `uint256`
   wrappers.

   Port `&`, `^`, `~`, and `|` as concrete 256-bit operations matching
   the C++ structure.

4. Revisit `signextend` if needed.

   If the first port uses shift-only identities, consider a second pass
   that rewrites it into a mask-valued form closer to the C++ if the
   new bitwise interface makes that substantially clearer.

5. Port `countr_zero` and `popcount`.

   These are the two utility functions most likely to benefit from new
   word-level primitives, because their control flow in C++ is simple
   and obvious, and a direct Rocq match is preferable to a bespoke
   bit-scan encoding.

6. Decide where serialization belongs.

   `load_be`, `load_le`, `store_be`, `store_le`, `from_bytes`, and
   string parsing are probably better in a dedicated byte or encoding
   module than in `Arithmetic.v`. Still mirror the C++ observable
   behavior exactly, especially padding and endianness.
