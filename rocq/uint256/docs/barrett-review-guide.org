* Purpose

This document is the single entry point for human review of the Barrett
reduction work in the uint256 Rocq development.  It consolidates the
review-facing material created so far from:

- [[file:barrett-cpp-correspondence.org][barrett-cpp-correspondence.org]]
- [[file:barrett-modeling-deviations.org][barrett-modeling-deviations.org]]
- [[file:barrett-proof-adjustments.org][barrett-proof-adjustments.org]]
- [[file:barrett-proof-obligations.org][barrett-proof-obligations.org]]
- [[file:barrett-semantic-discrepancies.org][barrett-semantic-discrepancies.org]]

The source documents are provenance logs.  This guide is organized by what a
new reviewer needs first: the current model/proof surface, the C++
correspondence, the known non-literal modeling choices, semantic caveats, and
the current proof status.

The main rule for the development is:

- if Rocq disagrees with the C++ implementation, update the Rocq model to
  match the C++;
- if the C++ implementation disagrees with the intended mathematical operation,
  record the discrepancy and add a concrete Rocq witness when practical.

* Quick Status

The current Barrett model is in [[file:../theories/Barrett.v][Barrett.v]].
The current proofs are in [[file:../theories/BarrettProofs.v][BarrettProofs.v]].

The current proof surface covers both reducer stacks:

- =reduce_correct_with_quotient=, used by =udivrem_correct= and
  =signed_wrapper_correct=;
- =low_product_sufficient=
- =reduce_correct_remainder_only=

The public wrapper theorem families currently proved are:

- =udivrem_correct= and the concrete =udivrem_reciprocal_*= alias
  corollaries;
- =signed_wrapper_correct=, excluding the standard =INT_MIN / -1= signed
  overflow case;
- =addmod_correct=
- =mulmod_correct=
- =mulmod_const_correct=
- the concrete =addmod_reciprocal_*=, =mulmod_reciprocal_*=, and
  =mulmod_const_reciprocal_*= alias corollaries.

Additional review-support files are:

- [[file:../theories/BarrettStaticChecks.v][BarrettStaticChecks.v]], which mirrors
  the C++ =static_assert= checks for concrete alias instantiations;
- [[file:../theories/BarrettReviewWitnesses.v][BarrettReviewWitnesses.v]], which
  contains executable review witnesses for resolved or non-exported generic
  concerns.

The most important current caveat is that the generic no-multiplier
remainder-only proof surface assumes a word-aligned post-product shift.  The
exported aliases satisfy this condition.  A non-exported generic parameter
witness shows that the template is not fully generic for unaligned
post-product shifts without either a C++ constraint or a wider
remainder-only product computation.

The main non-literal shift detail is the final-word branch of =rshift=.  The
C++ chooses between =shrd(high, low, bit_shift)= and =low >> bit_shift= based
on whether the high input word is in bounds.  The Rocq model uses total
zero-padded =get_word=, so the out-of-bounds branch is represented as
=shrd64 zero low bit_shift=.  This is equivalent at the integer level, but it
does not expose the C++ guard directly.

* Modeling Deviations

The deviations below are representational.  They are not claims that the C++
implementation has different arithmetic behavior.

** Fixed-size arrays and spans

C++ uses =words_t<N>= fixed-size arrays and =std::span= output slices.  Rocq
uses little-endian =words= lists, with size information tracked by derived
functions such as =input_words=, =reciprocal_words=, and =max_r_hat_words=.
Copies and truncations are modeled with =fit_words= rather than in-place span
writes.

** Template-time computation

C++ computes constants such as =INPUT_WORDS=, =RECIPROCAL_BITS=,
=PRE_PRODUCT_SHIFT=, and =RELEVANT_QUOTIENT_WORDS= at template-instantiation
time.  Rocq models them as ordinary pure functions over =BarrettParams=.
Likewise, =if constexpr= branches are represented as ordinary conditionals.

** Constructors and assertions

The Rocq constructors =reciprocal_of_denominator= and
=reciprocal_of_multiplier= are pure functions.  Debug assertions and template
requirements become theorem assumptions such as parameter admissibility,
denominator validity, and reciprocal admissibility.

** Wrapper result types

The Barrett =udivrem= and =sdivrem= wrappers reuse the existing
=Division.udivrem_result= type.  The modulo wrappers return =uint256=.  This
matches the rest of the uint256 Rocq development and does not change the
arithmetic contract.

** Dormant multiplier numerator branch

The generic C++ branch for =MULTIPLIER_WORDS > 0= and =BIT_SHIFT <> 0= appears
to be ill-formed as currently written, but exported constant-multiplier
aliases instantiate =INPUT_BITS = 256= and therefore =BIT_SHIFT = 0=.  The
Rocq model uses the intended word-shift behavior for =numerator_with_multiplier=
and theorems about the reachable constant-multiplier wrappers assume
=bit_shift params = 0=.

** =rshift= final-word branch

As noted above, total zero-padded word access compresses the C++ final-word
branch.  If future review requires a literal control-flow model here, prefer
adding a local partial accessor for =rshift= over changing =get_word=
globally.  The total accessor is used throughout the uint256 development to
model zero extension in addition, subtraction, multiplication, and division.

* Confirmed Semantic Differences from Textbook Algorithms

These are places where the Monad implementation is not a literal transcription
of Barrett 1986, HAC 14.3.3, or Modern Computer Arithmetic Section 2.4.1.
They are not necessarily bugs.

** Family of reciprocal reducers

The references usually fix one modulus =m=, one scale
=mu = floor(b^(2k) / m)=, and one quotient derivation.  The Monad code
parameterizes over denominator interval, input width, and optional baked-in
multiplier.  The proof target is therefore a family of reciprocals and
wrappers, not one monolithic textbook reducer.

** Pre-product shift

The implementation may replace =x= by =floor(x / 2^B)= before multiplying by
the reciprocal.  When =B = 0=, the quotient underestimate is at most one.
When =B > 0=, the proof target uses the weaker bound that the underestimate is
at most two.  This changes the proof shape but is consistent with the
implementation and the addendum report.

** Denominator 1

The exported alias families cover denominators in =[2, 2^256]=, not
=d = 1=.  If VM semantics need divisor or modulus =1=, that should be handled
by a trivial separate path rather than by the current Barrett alias families.

** Signed division wrapper

The signed wrapper is an unsigned Barrett call over absolute values followed
by sign restoration.  The proof excludes the standard two's-complement
=INT_MIN / -1= overflow case unless the intended API is later restated as a
wrapped signed division contract.

** Exact widened numerators

=addmod= constructs the exact 257-bit sum before reduction.  =mulmod=
constructs the exact 512-bit product.  =mulmod_const= stores one multiplicand
inside the reciprocal.  The natural theorems are therefore:

- =addmod(x, y, rec) = (x + y) mod d=
- =mulmod(x, y, rec) = (x * y) mod d=
- =mulmod_const(x, rec) = (x * y) mod d=

for all 256-bit inputs, not only for already-reduced residues.

** Fixed-width shift semantics

The interval upper bounds rely on =uint256_t= fixed-width shifts.  In
particular, =uint256_t{1} << 256= evaluates to zero, and subtracting one
wraps to =2^256 - 1=.  The Rocq alias model follows that helper-type behavior
rather than arbitrary-precision mathematical shift semantics.

* Current Caveats and Resolved Candidates

** Unused generic multiplier branch with nonzero bit shift

In =reciprocal::numerator(y)=, the generic =MULTIPLIER_WORDS > 0= and
=BIT_SHIFT <> 0= branch appears to read =y[i+1]= up to
=i = MULTIPLIER_WORDS - 1=.  That is one word past the end of =y=.

This branch is unreachable for exported multiplier aliases because they use
=INPUT_BITS = 256= and hence =BIT_SHIFT = 0=.  It is logged as outside the
verified public surface.

** Generic multiplier bit bound

The generic constant-multiplier constructor stores =y= in
=words_t<MULTIPLIER_WORDS>= but =MAX_QUOTIENT_BITS= is computed from
=MULTIPLIER_BITS=.  If =MULTIPLIER_BITS= is not a multiple of 64, the generic
template can admit multipliers too large for the quotient/product widths used
by =estimate_q=.

The exported =mulmod_const= aliases use =MULTIPLIER_BITS = 256=, so every
=uint256_t= multiplier satisfies the required bit bound there.  Generic
theorems record the required bound in =reciprocal_admissible=.

** Remainder-only quotient truncation candidate

A suspected mismatch in the =need_quotient = false= path was checked in
=BarrettReviewWitnesses.v=.  The first no-multiplier interval example keeps
two whole quotient words, and both the full and remainder-only estimates
produce congruent products modulo =MAX_R_HAT_WORDS=.  The general theorem
=low_product_sufficient= has since been proved under the exported aliases'
word-alignment assumptions.  This candidate should not be cited as a semantic
discrepancy.

** Signed =INT_MIN / -1=

The signed wrapper follows C++ control flow and cannot represent the exact
mathematical quotient =+2^255= in the signed 256-bit interpretation.  The
theorem =signed_wrapper_correct= excludes this case.  If the API later wants
wrapping signed division semantics, the theorem should be restated with a
wrapped quotient postcondition.

* Proof Adjustments and Strengthened Premises

Theorem statements were strengthened only where the extra assumptions match
C++ semantics or a documented semantic precondition.

** Parameter admissibility

The C++ template =requires= clause says:

- =uint256_t{Params.min_denominator} > 0=
- =uint256_t{Params.min_denominator} <= uint256_t{Params.max_denominator}=
- =Params.input_bits > 0=

The Rocq record =BarrettParams= is total, so constructor and wrapper theorems
carry =params_admissible= explicitly.

** Multiplier storage fit

The multiplier constructor stores =y= in a fixed
=words_t<MULTIPLIER_WORDS>= field.  Theorems that compare the stored
multiplier with the original =uint256_t= require the corresponding fit
condition, matching the C++ debug assertion that higher words are zero.

** Reachable multiplier branch

The constant-multiplier theorems assume =bit_shift params = 0= because the
reachable exported aliases have =INPUT_BITS = 256=.  Proving the dormant
generic nonzero-bit-shift branch would not currently prove the C++ control
flow.

** Reciprocal admissibility

Generic theorems over arbitrary Rocq =reciprocal= records require
=reciprocal_admissible=.  This captures facts that the C++ object layout and
constructors enforce implicitly: denominator range, stored array lengths,
multiplier fit, and reciprocal scale bounds.

** Proof-module dependencies

=BarrettProofs.MakeProofs= depends on existing division, runtime
multiplication, and arithmetic proof modules.  This is proof plumbing, not a
C++ semantic change.

** Word-aligned post-product shifts

=low_product_sufficient= and the generic remainder-only reducer theorem
require =post_product_bit_shift params = 0=.  This matches the current proof
surface and all exported alias families.  The next section records the generic
alignment issue in more detail.

* Pre-Product Shift Alignment Issue

The no-multiplier =PRE_PRODUCT_SHIFT= calculation is intended to choose a
shift =k= such that:

- =k <= MIN_DENOMINATOR_BITS - 1=, preserving the quotient-error bound;
- =(SHIFT - k) mod 64 = 0=, making the post-product shift word-aligned.

The C++ adjustment subtracts =max_pre_product_shift_64 - BIT_SHIFT= only when
=max_pre_product_shift_64 > BIT_SHIFT=.  This preserves the exported interval
behavior, but it is not a fully generic computation of the largest aligned
=k=.  For example, the non-exported witness in
[[file:../theories/BarrettReviewWitnesses.v::108][UnalignedPostProductShift]]
has:

- =INPUT_BITS = 34=, so =BIT_SHIFT = 34=;
- =MIN_DENOMINATOR = 2=, so =MIN_DENOMINATOR_BITS - 1 = 1=;
- =PRE_PRODUCT_SHIFT = 1=;
- =POST_PRODUCT_BIT_SHIFT = 33=.

Both =k = 0= and =k = 1= leave a non-word-aligned post-product shift.  This is
outside the exported aliases, but it shows that the generic template does not
itself guarantee the current Rocq precondition.

There are two plausible repair strategies if unaligned generic intervals ever
matter.

1. Support unaligned post-product shifts.

   The remainder-only path currently sizes the pre-shift product using
   =RELEVANT_QUOTIENT_BITS + POST_PRODUCT_SHIFT=.  For unaligned shifts this
   can omit high bits that later shift down into the padding part of the final
   retained quotient word.  The C++ fix would size the remainder-only product
   by full quotient words:

   #+BEGIN_SRC c++
   constexpr size_t quotient_bits =
       need_quotient ? MAX_QUOTIENT_BITS
                     : 64 * RELEVANT_QUOTIENT_WORDS;
   constexpr size_t prod_bits = quotient_bits + POST_PRODUCT_SHIFT;
   #+END_SRC

   The Rocq model would factor the analogous helper:

   #+BEGIN_SRC coq
   Definition quotient_product_bits params need_quotient :=
     if need_quotient then
       max_quotient_bits params
     else
       word_width * relevant_quotient_words params.
   #+END_SRC

   Then =estimate_q false= should be provable congruent to the full quotient
   estimate modulo =2^(64 * relevant_quotient_words params)= without assuming
   =post_product_bit_shift = 0=.

2. Constrain the C++ generic template to the current proof surface.

   The current proof dependency is narrow.  =reduce_correct_with_quotient=,
   and therefore =udivrem= and =sdivrem=, do not require
   =post_product_bit_shift = 0=.  The alignment premise enters through
   =low_product_sufficient=, which is used by =reduce_correct_remainder_only=
   and therefore by =addmod=, =mulmod=, and =mulmod_const=.

   The clean C++ constraint is a guarded assertion inside =reduce=:

   #+BEGIN_SRC c++
   if constexpr (!need_quotient) {
       static_assert(POST_PRODUCT_BIT_SHIFT == 0);
   }
   #+END_SRC

   This preserves full-quotient generic =reduce<true>= instantiations while
   requiring the word-alignment condition exactly where the current
   remainder-only proof uses it.  Wrapper-level assertions in =addmod=,
   =mulmod=, and =mulmod_const= would protect the public wrappers but would
   still permit direct internal calls to =reduce<false>= with unaligned
   parameters.

* Static Checks and Executable Witnesses

[[file:../theories/BarrettStaticChecks.v][BarrettStaticChecks.v]] is the place
for Rocq equivalents of C++ =static_assert= facts over concrete aliases.  It
currently covers reciprocal bit widths, maximum denominator bit widths,
selected pre-product word shifts, and the addmod catch-all =input_words=
assertion.

[[file:../theories/BarrettReviewWitnesses.v][BarrettReviewWitnesses.v]] is the
place for executable review witnesses.  It is not a file of counterexamples
to C++ correctness.  Current witnesses include:

- the resolved remainder-only quotient-truncation candidate;
- the non-exported unaligned post-product-shift example.

If a future semantic blocker is suspected, first decide whether it is a C++
incorrectness issue or only a Rocq/C++ mismatch.  Only C++ incorrectness is a
semantic blocker.  Rocq/C++ mismatches should be fixed in the model.

* Review Checklist

Use this order for side-by-side review:

1. Check that derived constants in =Barrett.v= match the corresponding
   =static constexpr= formulas.
2. Check that =numerator= and =numerator_with_multiplier= match the reachable
   C++ constructor behavior.
3. Check =rshift= using the final-word branch deviation note.
4. Check that =estimate_q= preserves the pre-shift, product, and post-shift
   structure.
5. Check that =reduce= preserves the provisional remainder, correction order,
   and quotient update branches.
6. Check wrapper inputs: 256-bit =udivrem=, 257-bit =addmod=, 512-bit
   =mulmod=, and 256-bit online input for =mulmod_const=.
7. Check that every generic theorem premise corresponds either to C++
   template/object semantics or to a documented precondition.
8. Check that any new suspected C++ semantic blocker has a concrete Rocq
   witness before it is treated as blocking proof work.
